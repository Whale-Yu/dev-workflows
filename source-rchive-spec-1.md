# 个人源码资产归档与命名规范手册 (V1.0)

本规范旨在建立一套结构清晰、技术安全、可追溯且具备工业级素养的代码归档体系。通过规范化管理旧代码与新项目，实现资产的快速检索与技术沉淀。

---

## 一、 核心命名公式

所有项目根目录文件夹命名必须严格遵循以下三段式结构：


```
[8位日期]_ [4位分类前缀]_[全小写烤肉串项目名]
```

### 格式定义：

* **8位日期 (`YYYYMMDD`)**：项目启动或主要成果产出的具体日期。
* **4位分类前缀 (`Type_`)**：首字母大写，后接下划线 `_` 作为强隔离。
* **全小写烤肉串项目名 (`kebab-case`)**：一律使用小写英文单词，单词之间用横线 `-` 弱分隔。

> **正确示例**：`20260524_Proj_crawler-leafly-data`
> **错误示例**：`202605_proj_Crawler_LeaflyData`（日期不全、前缀未大写、项目名混用大驼峰和下划线）

## 二、 分类前缀定义 (The 7 Pillars)

为了兼顾技术深度与业务场景，将所有代码资产划分为以下 7 大核心维度：

| 前缀 | 全称 | 定义与适用场景 | 含金量 / 隐私级别 |
| --- | --- | --- | --- |
| **`Learn_`** | Education | 跟着教程走、基础语法练习、新框架 Demo、LeetCode 算法刷题。 | 基础积累 / 完全公开 |
| **`Proj_`** | Project | 个人独立发起、具有完整前后端逻辑或独立功能的**实战全栈/算法应用**。 | 核心作品 / 完全公开 |
| **`Cmp_`** | Competition | 各类黑客马拉松、数学建模、AI算法赛、机器人大赛等**竞赛代码**。 | 亮点成果 / 完全公开 |
| **`Grad_`** | Graduation | 阶段性结业的核心项目、**毕业设计**、学术论文配套的对比实验代码。 | 里程碑 / 谨慎公开 |
| **`Work_`** | Work / Biz | 公司实习、导师横向课题、商业外包等**具有商业机密属性的工业级代码**。 | 商业履历 / **严禁公开** |
| **`Tool_`** | Utility / Tool | 自己写来提高日常效率的提效脚本、自动预约工具、爬虫小工具。 | 实用工具 / 视情况公开 |
| **`Misc_`** | Miscellaneous | 期末大作业碎文件、临时想法沙盒、写了十几行就废弃的临时垃圾箱。 | 杂项堆放 / 定期清理 |

## 三、 视觉与技术排版规范

### 1. 符号分工原则

* **下划线 `_`（强隔离）**：仅用于分隔**时间、分类、项目名**这三个不同的维度。
* **连字符 `-`（弱分隔）**：仅用于分隔项目名内部的**多个英文单词**。

### 2. 为什么坚持“全小写 + 横线 (`kebab-case`)”？

* **跨平台安全**：Windows/macOS 对大小写不敏感，而 Linux 服务器和 Git 底层严格敏感。全小写可彻底杜绝跨平台协作时的路径找不到、Git 无法追踪大小写变更等史诗级大坑。
* **URL 天然友好**：项目名直接对应 GitHub 仓库名或 Web 部署路径，小写加横线更符合现代 Web 标准。
* **高可读性**：在代码编辑器（如 VS Code）或终端中，双击横线连接的单词可以精准选中单词，而不会连带选中分类前缀。

## 五、 资产上云（GitHub）防火墙原则

在将本地归档代码推送到 GitHub 等云端托管平台时，必须严格遵守以下红线：

1. **Work 级别红线**：所有带 `Work_` 前缀的文件夹，**严禁**上传至公开仓库。如需备份，必须建立私有仓库（Private Repository）。
2. **敏感信息审查**：所有项目在推送到远端前，必须检查代码内是否硬编码了个人真实密码、数据库连接串、阿里云/腾讯云的 `AccessKey`、企业内部 API 接口。
3. **大文件过滤**：在项目根目录配置 `.gitignore`，提前过滤掉 `.venv/`、`node_modules/`、以及 `*.pt`、`*.onnx` 等大于 100M 的模型权重文件。
