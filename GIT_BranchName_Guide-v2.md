## 🏆 专家级 Git 分支命名标准 (v2.0)

本规范遵循 **SEMVER (语义化版本)** 逻辑：`主版本号 (Major) . 次版本号 (Minor) . 修订号 (Patch)`。

### 一、 分支命名核心：看“大版” (`X.Y`)

分支代表一条**持续演进的生产线**。通常不需要为每一个三级补丁号（Patch）创建独立分支。

| 分支名称示例 | 适用场景 | 对应语义 |
| --- | --- | --- |
| **`main`** | 永远指向当前最稳定的生产版本。 | 对应所有正式 Tag |
| **`3.x-dev`** | **3.x 系列的主开发分支**。承载 3.3.0, 3.3.1, 3.3.2 等所有小版本的演进。 | 次版本号级别 (Minor) |
| **`feature/xxx`** | 具体的某个功能模块开发。 | 功能点级别 |
| **`hotfix/xxx`** | 紧急修复 `main` 上的线上故障。 | 修订号级别 (Patch) |

---

### 二、 补丁管理核心：看“标签” (`X.Y.Z`)

三级版本号（Patch）的标记任务交给 **Git Tag**。它是“死板的快照”，最适合记录 `3.3.1` 这种静止的时间点。

#### 操作流转逻辑：

1. **在 `3.x-dev` 持续工作**：修复 3.3.0 的小 Bug 或优化小功能。
2. **达到交付点**：当积累了一定量的小改动，准备发布 `3.3.1`。
3. **合并至 `main**`：将 `3.x-dev` 合并入主分支。
4. **盖章归档 (关键)**：在 `main` 上打上 `v3.3.1` 的标签。
* `git tag -a v3.3.1 -m "fix: 优化了座位匹配算法"`



---

### 三、 特殊情况：何时需要 `3.3.x-fix` 分支？

只有在一种情况下，你需要细分出三级版本分支：**“多版本并行维护”**。

* **场景**：你已经在 `3.4-dev` 上开发新功能了，但旧的 `3.3` 用户反馈了一个严重 Bug，你必须为 `3.3` 发布一个 `3.3.1` 补丁。
* **做法**：
1. 从 `main` 的 `v3.3.0` 标签处拉出一个临时分支：`fix/3.3.1-patch`。
2. 修复完成后，合并回 `main` 并打标 `v3.3.1`。
3. **同时**，将该修复合并回你现在的 `3.4-dev`，确保新版也补上了这个洞。



---

### 四、 完整命名指南清单

| 分类 | 命名建议 | 说明 |
| --- | --- | --- |
| **主干** | `main` | 唯一、受保护、最稳定。 |
| **开发基地** | `3.x-dev` | **推荐**。如 `3.3-dev`。覆盖该版本下的所有小补丁。 |
| **功能开发** | `feature/id-desc` | 例如 `feature/qr-scanner`。 |
| **重构优化** | `refactor/desc` | 例如 `refactor/cleanup-3.2-json`。 |
| **版本快照** | `vX.Y.Z` (Tag) | **必须**。例如 `v3.3.0`, `v3.3.1`。 |

---

### 💡 总结与建议

对于你的 `SmartLib` 项目，**“分支看大版，Tag 控小版”** 是最省力且最高效的方案。

* **平时：** 只管在 `3.3-dev` 提交。
* **发布：** 只要代码进了 `main`，就立刻打一个 `v3.3.x` 的 Tag。
