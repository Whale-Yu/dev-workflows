# 🚀 Github 指南

## 一、 分支操作 (Branch) —— 动态的进度条

分支用于在不影响主线的情况下进行迭代。分支指针会随 `push` 自动向前漂移。

### 1. 基础管理

| 操作        | 命令                       | 说明            |
| :-------- | :----------------------- | :------------ |
| **查看分支**  | `git branch -a`          | 查看本地和远程所有分支状态 |
| **创建分支**  | `git branch <name>`      | 基于当前节点新建分支    |
| **切换分支**  | `git checkout <name>`    | 跳转到指定分支       |
| **创建并切换** | `git checkout -b <name>` | 新建并立即跳转（常用）   |

### 2. 远程同步

- **同步远程列表**：`git fetch origin` (当远程多了 `main` 但本地看不到时执行)
- **关联远程分支**：`git checkout -b main origin/main`
- **推送分支**：`git push origin <branch_name>`

## 二、 标签操作 (Tag) —— 凝固的快照

标签用于标记特定版本，一旦打上，除非手动删除，否则永远指向该提交。


> **⚠️ 排序准则**：在 Git 和包管理工具中，`v1.0.0-rc` < `v1.0.0`。带有预发布后缀的版本被视为“不完整”的版本。

| 分类 | 后缀名称 | 全称 / 含义 | 稳定性 | 适用场景与面向对象 | 示例 |
| :--- | :--- | :--- | :--- | :--- | :--- |
| **预发布**<br>*(Pre)* | **`-alpha`** | Alpha / 内测版 | 🔴 极低 | **内部开发阶段**。功能不全或存在严重 Bug，仅供开发人员调试。 | `v1.0.0-alpha.1` |
| | **`-beta`** | Beta / 公测版 | 🟡 中等 | **外部测试阶段**。功能已定型，通过社区反馈收集边界 Bug。 | `v1.0.0-beta.2` |
| | **`-rc`** | Release Candidate<br>发行候选版 | 🟢 极高 | **准正式阶段**。若无重大 Bug 汇报，此版本将直接转化为正式版。 | `v1.0.0-rc.1` |
| **正式版** | *(无后缀)* | **Stable Release** | 🔵 完美 | **标准生产环境**。已经过充分验证，推荐所有用户使用的正式版本。 | `v1.0.0` |
| **维护期**<br>*(Post)* | **`-stable`** | Stable / 稳定版 | 🔵 完美 | **里程碑标识**。版本已平稳运行（如2个月以上），强调极其可靠。 | `v1.0.0-stable` |
| | **`-lts`** | Long Term Support<br>长期支持版 | 🔵 完美 | **企业级选择**。承诺未来 1-3 年内会持续提供安全更新。 | `v2.0.0-lts` |
| | **`-ga`** | General Availability<br>正式商用版 | 🔵 完美 | **商业发布**。经过严格合规性检查后的最终交付版本。 | `v1.5.0-ga` |
| **已废弃** | **`-deprecated`** | **Deprecated Release** | ❌ 废弃 | **已废弃的版本** | `v1.0.0-deprecated` |


### 💡 标签命名最佳实践建议：

1.  **平稳运行后的晋升**：如果 `v1.2.0-rc.2` 在生产环境平稳运行了 2 个月，**最规范的做法**是打一个新的标签 `v1.2.0`（去掉所有后缀），代表它正式晋升为稳定版。
2.  **增量标记**：如果同一阶段有多次更新，请在后缀后加数字，如 `v1.0.0-beta.1` -> `v1.0.0-beta.2`。
3.  **小写原则**：后缀建议全部使用小写字母。

### 1. 基础管理

| 操作         | 命令                          | 说明              |
| :--------- | :-------------------------- | :-------------- |
| **打附注标签**  | `git tag -a v1.0 -m "说明"`   | 创建一个带备注的版本号（推荐） |
| **查看标签**   | `git tag`                   | 列出项目所有版本号       |
| **推送标签**   | `git push origin <tagname>` | 将版本号同步到 GitHub  |
| **推送所有标签** | `git push origin --tags`    | 一次性同步所有本地版本号    |

### 2. Release 发布规范

当 Tag 推送到 GitHub 后，建议在 GitHub 网页上创建对应的 Release，包含以下要素：

#### 📝 Release Title（标题）

建议格式：`vX.Y.Z - 简短描述`

- `v1.0.0 - 初始版本发布`
- `v1.0.1 - 修复登录超时 Bug`
- `v1.1.0 - 新增数据导出功能`
- `v2.0.0 - 重构核心模块，API 不兼容升级`

#### 📋 Release Notes（发布说明）

建议包含以下内容：

```markdown
## 🎉 发布概要
一句话简述本次发布的主要内容。

## ✨ 新功能
- 功能 A 的描述
- 功能 B 的描述

## 🔄 优化
- 优化了性能 1
- 优化了性能 2

## 🐛 Bug 修复
- 修复了问题 1
- 修复了问题 2

## ⚠️ 破坏性变更（仅 vX.0.0 需要）
- API 变更 1
- API 变更 2

## 📦 下载链接
- Windows: [下载链接]
- macOS: [下载链接]
- Linux: [下载链接]
```

#### 📎 附件（Assets）

建议上传以下文件（根据项目类型）：

- **编译好的可执行文件**（Windows `.exe`、macOS `.app`、Linux 二进制）
- **源码压缩包**（GitHub 会自动生成，也可手动上传）
- **安装包**（`.deb`、`.rpm`、`.dmg` 等）
- **文档**（PDF 格式的使用说明）
- **示例数据**（测试用的数据集）

**建议：** Release Notes 可以通过 Git Commit 记录自动生成（使用工具如 `standard-version`、`release-please` 等）。

## 三、 Release (发布) 与 Tag 的联动

Release 是 GitHub 基于 Tag 提供的可视化发布功能，用于存放编译好的文件或源码包。

### 💡 核心逻辑：动静之别

1. **分支是流动的**：当你向 `smartlib2.0` 推送代码，分支指针会移动到新位置。
2. **标签是死板的**：Tag 依然留在原地，不会随推送更新。
3. **Release 是锁定的**：Release 绑定的是 Tag。因此，**如果你只推送分支而不更新 Tag，Release 里的内容将永远是旧的。**

**更新 Release 的正确姿势：**

- **方案 A（规范）**：修复 Bug 后，打一个新标签（如 `v2.1`）并推送，然后在 GitHub 上发一个新 Release。
- **方案 B（覆盖）**：删除远程旧 Tag，在最新提交上重打同名 Tag 并推送，最后在 GitHub Release 页面编辑重新关联。

***

### 📌 什么时候需要发布 Tag 和 Release？

**简短答案：只要是对外发布的版本，都建议打 Tag 并发布 Release。**

#### 具体情况分析：

| 版本变更            | 类型          | 是否打 Tag | 是否发 Release |
| --------------- | ----------- | ------- | ----------- |
| v1.0.0 → v1.0.1 | 修订号（Bug 修复） | ✅ 建议    | ✅ 建议        |
| v1.0.2 → v1.1.0 | 次版本号（新增功能）  | ✅ 必须    | ✅ 必须        |
| v1.3.2 → v2.0.0 | 主版本号（不兼容变更） | ✅ 必须    | ✅ 必须        |

#### 实际发布建议：

| 场景       | 是否打 Tag | 是否发 Release |
| -------- | ------- | ----------- |
| 内部开发阶段   | 可选      | 可选          |
| 提交给测试/客户 | ✅ 是     | ✅ 是         |
| 正式上线     | ✅ 是     | ✅ 是         |
| 修复线上 Bug | ✅ 是     | ✅ 是         |

#### 大版本更新的分支策略（v2.0 示例）：

```
main (稳定版，打 v1.3.2 Tag)
  └── dev-v2.0 (开发分支，进行不兼容的重构)
       └── 开发完成后合并回 main
           └── 打 v2.0.0 Tag 并发布 Release
```

#### 实际项目分支策略示例（多版本并行）：

假设你的项目有以下分支：

- `main`：默认分支，不存放代码（仅用于版本发布整合）
- `smartlib1.0`：v1 版本稳定分支
- `smartlib2.0`：v2 版本稳定分支
- `smartlib`：当前开发分支（v3 版本，可能有不兼容变更）

**推荐策略：**

```
main (不存放代码，仅作为发布整合点)
│
├── smartlib1.0 (v1 稳定版，打 v1.0.0、v1.0.1 等 Tag)
│   └── 仅修复 Bug，不新增功能
│
├── smartlib2.0 (v2 稳定版，打 v2.0.0、v2.1.0 等 Tag)
│   └── 新增功能和 Bug 修复，向下兼容
│
└── smartlib (v3 开发版，进行不兼容的重构)
    └── 开发完成后创建新分支 smartlib3.0
        └── 打 v3.0.0 Tag 并发布 Release
```

**工作流说明：**

1. **v1/v2 稳定分支**：只在必要时修复 Bug，用修订号递增（v1.0.1、v2.0.1）
2. **smartlib 开发分支**：进行 v3 的激进开发和重构
3. **发布时**：在对应的稳定分支上打 Tag 并发布 Release
4. **v3 完成后**：从 smartlib 创建新的稳定分支 smartlib3.0
